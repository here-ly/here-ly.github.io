第一章：认识 Markdown —— 为什么它值得学习

1.0 写在前面

笔者于今天开始写博客，还没开始写，就在 md 编辑器决定上思考了半天，于是我觉得用 md 入门作为我的第一篇技术博客。

这篇技术博客会包含以下内容：

  • md 的用途
  • 部分教程以及评价
  • md 编辑器横评

假如你是刚刚学习计算机的小白，学好 md 可以大大增强你的学习和交流能力。

假如你是正在优化 harness 的 vibe coder，对 md 更灵活的使用，无疑是理想的 agent 输入。

假如你正在筹备自己的博客，一个合理的 md 编辑器自然是你需要的。

假如你已经通晓万物，希望你可以指出我的不足，让我对我的博客进行改进。

1.1 Markdown 是什么

是这篇博客使用的语言

2004 年，程序员 John Gruber 做了一件看似微不足道、却影响深远的事：他创造了一种让普通人也能轻松排版文字的工具，并把它命名为 Markdown。

在 Markdown 出现之前，如果你想在电脑上写一段格式漂亮的文字，通常只有两条路可走：

  • 走进 Word 的世界：调整字体、设置行距、对齐段落……你花在排版上的时间，往往比写作本身还多。更糟的是，换一台电脑，格式可能全乱。

    丑陋而糟糕的闭源语言

  • 拥抱 HTML 的严谨：用 <h1> 写标题，用 <strong> 加粗，用 <a> 插入链接。功能强大，但满屏的标签让阅读原始文件变成一场灾难。

    实则笔者还没学会

Markdown 走了第三条路。它用几个简单到近乎直觉的符号，就能表达丰富的格式：

渲染后的效果美观大方，而更重要的是——即使不渲染，纯文本本身也极易阅读。这正是 Gruber 的设计哲学：“Markdown 的语法应该让人在作为纯文本阅读时，也能清晰传达结构和语义。”

对于 agent，这一点十分重要，最近虽然有不少怀疑 html 会取代 md 的声音，我对 html 取代 md 一直持怀疑态度，诚然 html 更加好看严谨，但 md 更接近 llm 预训练语料的，llm 在这类文本内会有更加的表现，html 可能会造成不必要的上下文污染。

1.2 Markdown 有哪些优势

纯文本的永恒性

想象一下，多年后，你翻开今天写的日记。如果是 Word 文档，你可能需要先找到一台还能运行旧版 Office 的电脑；如果是某个笔记软件的专属格式，而这家公司已经倒闭，你的文字可能永远锁在加密文件里。闭源软件全责

但 Markdown 不同。它是纯文本。五十年前诞生的 .txt 文件今天依然能打开，五十年后的 Markdown 文件也一样。不依赖任何特定软件，不担心格式过时，这是数字时代最接近"纸和笔"的永恒性。

版本控制友好

对于写作者来说，"改稿"是常态。传统的方式是保存 文章_v1.docx、文章_v2.docx、文章_最终版.docx、文章_最终版_真的.docx，最后文件夹里堆满了谁也分不清的版本。

如果你使用 Git，就会理解部分 Markdown 的好处：每一次修改都被记录，你可以看到"第三段被删除了"、“哪个链接被替换了”、“什么时候加了注释”。对于技术文档、学术论文、甚至是小说创作，这种可追溯性都是无价之宝。

跨平台通用

Windows、macOS、Linux、iOS、Android——无论你用哪种设备，都有无数款支持 Markdown 的编辑器。你今天在 Mac 上写的文章，明天在 Windows 上打开，后天在手机上修改，格式不会有丝毫错乱。

更进一步，几乎所有现代内容平台都原生支持 Markdown：GitHub、Notion、语雀、知乎、CSDN……学会一套语法，到处都能用。

AI 时代的结构化优势

这是 Markdown 在 2020 年之后获得的新优势。当你与 ChatGPT、Claude、Gemini 等大语言模型交流时，会发现一个有趣的现象：这些 AI 对 Markdown 的理解，远比对 Word 文档或富文本的理解要准确得多，而且它们的输出也是 md 格式。

这个选择就来源于 md 的泛用性，用户可以把回答复制到任何一个现代软件并保持原样的同时，可以输出公式、层次、表格等复杂结构。

用代码块包裹指令，AI 能精准识别边界；用表格组织需求，AI 能按列对齐输出；用层级标题梳理逻辑，AI 的回答也会更有条理。在 Prompt Engineering 中，Markdown 几乎成了一种通用语。

专注内容，而非排版

这可能是 Markdown 最被低估的优点。当你用 Markdown 写作时，你不会（其实是没法）纠结"这个标题用几号字"、“这段引用要不要缩进两字符”、“列表项的圆点是实心还是空心”。

你只有一个任务：把符号放在正确的位置，然后继续写下去。格式不会打断你的思路，排版不会消耗你的意志力。对于创作者来说，这种"无摩擦"的体验，本身就是一种奢侈。这就是为什么我的博客不美观，我会去学其它语言的，也许。

第二章：Markdown 的六大战场

讲了 Markdown 的理念和优势，接下来看看它到底能在哪些场景下大展身手。

2.1 技术文档与 README

这是 Markdown 的"发家之地"。打开 GitHub，几乎每个项目根目录下都有一个 README.md（这里 md 并不是粗口，而是 markdown 的标志）。开发者用它来介绍项目功能、展示安装步骤、列出使用示例、说明贡献规范。

为什么程序员爱它？因为技术文档需要频繁更新、多人协作、版本对齐。Markdown 让这一切变得简单：提交一次代码，文档同步更新；发起一个 Pull Request，评审者直接在行内评论；文档和代码放在同一个 Git 仓库里，永远不会"文档说 A，代码跑 B"。

2.2 个人博客与文章

越来越多的独立博客作者选择用 Markdown 写作。静态网站生成器（如 Hugo、Jekyll、Hexo）直接把 .md 文件转换成精美网页；Ghost、WordPress 等平台也提供了 Markdown 编辑模式。

对于写作者来说，这意味着内容即资产。你的文章不锁在某个平台的编辑器里，而是实实在在保存在本地的 .md 文件中。想换博客主题？直接迁移。想备份到网盘？复制粘贴即可。

2.3 笔记与知识管理

Obsidian、Logseq、Joplin、思源笔记……这些近年来的知识管理工具，底层都是 Markdown。你可以用 [[双链]] 连接相关概念，用标签 #tag 组织主题，用代码块记录技术片段，用任务列表管理待办。

笔者推荐 obsidian，以后会有相关博客。

在这个场景下，Markdown 不仅仅是一种格式，更是构建第二大脑的基础设施。你的知识以纯文本形式沉淀，十年后再翻开，依然鲜活可读。

2.4 AI 对话与 Prompt 编写

当你想让 AI 帮你写代码、改文章、分析数据时，用 Markdown 组织你的 Prompt，往往能得到质量高得多的回答。

举个例子：如果你直接说"给我写个 Python 脚本"，AI 可能给出一段格式混乱的代码；但如果你用 Markdown 写成：

AI 会立刻明白你的层次结构，输出的代码格式工整、注释清晰、逻辑完整。在和 LLM 协作的时代，Markdown 是效率的放大器。

2.5 演示幻灯片

别再用 PowerPoint 的拖拽和动画折磨自己了。如果你做的是技术分享、学术报告、或者任何"内容大于形式"的演示，Marp、Reveal.js、Slidev 等工具让你直接用 Markdown 写幻灯片。

每一页幻灯片就是几个标题和几行正文，逻辑清晰、修改便捷。你不需要花时间对齐文本框、调整配色方案，只需要关心"这一页要传达什么信息"。对于开发者和技术写作者来说，这是一种解脱。

2.6 学术论文与书籍

这听起来可能有些意外——Markdown 不是学术圈的"正经工具"。但借助 Pandoc，你可以把 Markdown 转换为 LaTeX、PDF、Word、ePub 等多种格式，完美处理公式、脚注、文献引用、目录生成等需求。

很多学者已经开始用 Markdown 写初稿：它足够轻量，让你快速推进思路；它又能导出成期刊要求的格式，不需要在写作阶段就陷入 LaTeX 的复杂语法。对于长篇著作（如技术书籍），Markdown 配合版本控制，比 Word 的"修订模式"优雅太多。

第三章：上手 Markdown —— 工具准备与学习路径

3.1 推荐你的第一个练习环境

如果你还没有安装任何 Markdown 编辑器，我建议从 VS Code 开始。它是免费的、跨平台的，而且装上 Markdown 插件后，写 Markdown 的体验会非常流畅。

下载与安装：

  • VS Code 官方下载页面 — 选择对应系统的版本安装即可

安装 Markdown 插件：

  • Markdown 插件推荐与使用指南 — VS Code 官方文档，介绍内置的 Markdown 支持以及如何安装增强插件（如 Markdown All in One）

装好 VS Code 后，新建一个文件，把后缀改成 .md，就可以开始写了。按 Ctrl+Shift+V（Mac 是 Cmd+Shift+V）可以实时预览渲染效果。

如果你暂时不想安装软件，或者想先在线体验一下，可以用这两个工具：

  • StackEdit — 在线 Markdown 编辑器，左边写、右边实时预览，不需要注册就能用，适合第一次上手时随便试试
  • Dillinger — 另一个在线编辑器，界面简洁，支持导出 PDF 和 HTML，如果你想在线看看 Markdown 渲染出来的实际效果，用这个很方便

3.2 按用途选择你的学习教程

以下教程按使用场景排列，你可以直接跳转到最符合你需求的章节。

场景一：技术文档与开发写作

教程：VS Code Markdown 支持 - 官方文档

用途评价：适合程序员和技术写作者。如果你已经在用 VS Code 写代码，不需要切换工具就能写专业的技术文档。这篇官方教程会教你如何在编辑器中预览 Markdown、使用快捷键、以及推荐哪些插件能大幅提升写作效率。学完后，你可以直接开始写项目的 README 或技术文档。

场景二：通用入门与日常写作

教程：Markdown Guide - Getting Started

用途评价：适合所有新手。这是最权威的 Markdown 入门教程，从最基础的语法开始，循序渐进地讲解标题、段落、列表、链接、图片、表格等核心元素。无论你是想写博客、做笔记，还是整理资料，学完这篇都能应对绝大多数日常场景。

场景三：动手实践与交互学习

教程：Markdown Tutorial

用途评价：适合喜欢边学边练的用户。这是一个在线交互式教程，每一课都会给你一个具体的写作任务，左侧输入 Markdown，右侧实时看到渲染效果。相比纯阅读式学习，这种方式印象更深。大约 30 分钟可以完成全部课程，之后你就能熟练写出格式漂亮的 Markdown 文档。

场景四：探索更多资源

教程：Awesome Markdown

用途评价：适合想要深入挖掘 Markdown 生态的进阶用户。这个 GitHub 仓库汇集了 Markdown 相关的编辑器、工具、库、教程等海量资源。当你掌握了基础语法，想要寻找特定场景的工具时，这里是绝佳的起点。

学习建议：如果你是第一次接触 Markdown，建议按 3.2（场景二）→ 3.2（场景三）→ 3.2（场景一）的顺序学习——先通过 Markdown Guide 建立整体认知，再用 Markdown Tutorial 动手巩固，最后根据你的实际工作场景深入学习 VS Code 的相关配置。

3.3 Markdown 的主要方言

当你开始搜索 Markdown 教程时，可能会遇到 “GFM”、“CommonMark”、“Obsidian Flavored Markdown” 之类的术语。这些都是 Markdown 的方言（Flavor）——它们在基础语法上大同小异，但在扩展功能上各有侧重。

了解方言很重要，因为：

  • 你选择的编辑器支持哪种方言，决定了你能使用哪些高级功能
  • 你写的文档要在哪里展示（GitHub、博客、笔记软件），那里的渲染器可能只支持特定方言
  • 不同方言之间不完全兼容，混用可能导致格式错乱

以下是四种最主要的方言及其核心特性：

CommonMark —— 标准规范

CommonMark 是 Markdown 的标准化版本，由一组开发者于 2014 年发起，旨在消除原始 Markdown 规范中的模糊地带。

主要特性：

  • 最严格的语法规范，定义清晰、无歧义
  • 只包含最基础的元素：标题、段落、列表、链接、图片、代码块、引用、分隔线
  • 不支持：表格、任务列表、删除线、脚注、数学公式等扩展功能

适用场景：

  • 追求最大兼容性的写作
  • 作为学习 Markdown 的"纯净起点"
  • 需要确保文档在任何平台都能正确渲染

代表平台： 严格遵循 CommonMark 的解析器、部分静态网站生成器

GFM（GitHub Flavored Markdown）—— 开发者首选

GFM 是 GitHub 在 CommonMark 基础上扩展的方言，是目前使用最广泛的 Markdown 变体。

官方文档与教程：

  • GFM 官方规范 — GitHub 官方发布的完整规范文档，适合查阅具体语法的精确定义

  • GitHub Markdown 交互教程 — GitHub 官方帮助文档，包含大量实时可交互的语法示例

    这里理应有一个交互网站，我还没有找到，等我学明白md了自己写一个

主要特性：

  • 完全兼容 CommonMark 的所有基础语法
  • 表格：用 | 和 - 绘制对齐表格
  • 任务列表：- [ ] 和 - [x] 创建可勾选的待办事项
  • 删除线：~~文字~~ 添加删除线效果
  • 代码块高亮：支持指定语言，获得语法着色
  • 自动链接：URL 无需包裹尖括号即可自动识别
  • @提及 和 Issue/PR 引用：#123、@username

适用场景：

  • GitHub 仓库的 README、Wiki、Issue、PR 描述
  • 技术文档和开发者博客
  • 初学者入门（兼容性好、功能实用）

代表平台： GitHub、GitLab、Gitee、VS Code 默认预览、Joplin、多数技术博客平台

Obsidian Flavored Markdown（OFM）—— 知识管理专用

Obsidian 是近年来最流行的个人知识管理工具之一，它在 GFM 基础上增加了大量面向知识工作者的功能。

主要特性：

  • 继承 GFM 的所有功能（表格、任务列表、代码高亮等）
  • 双向链接：[[笔记名]] 在笔记之间建立双向关联
  • 标签系统：#标签名 快速分类和检索
  • 脚注：[^1] 插入脚注，适合学术写作
  • LaTeX 数学公式：
    • 行内公式：用单个美元符号包裹，如 $E=mc^2$，会嵌入在段落文字中间
    • 块级公式（独立公式）：用双美元符号包裹，如 $$\sum_{i=1}^{n} x_i = X$$，会单独占一行并居中显示
    • 这是 OFM 的核心学术功能，支持完整的 TeX 数学语法，包括希腊字母、分数、矩阵、积分等
  • Mermaid 图表：直接在文档中绘制流程图、时序图、甘特图
  • Callouts：> [!note] 创建醒目的提示框（警告、提示、引用等）
  • 嵌入文件：![[其他笔记]] 将另一篇笔记的内容嵌入当前文档
  • YAML Front Matter：文档顶部的元数据区域，用于管理属性

适用场景：

  • 个人知识库和第二大脑构建
  • 学术研究（公式 + 脚注 + 文献引用）
  • 复杂项目的文档组织（用双向链接关联需求、设计、代码）

代表平台： Obsidian、部分支持双向链接的笔记软件

Pandoc’s Markdown —— 学术与多格式出版

Pandoc 是一个强大的文档转换工具，它的 Markdown 方言专为"写一次，到处发布"设计。

主要特性：

  • 包含 CommonMark + GFM 的大部分基础功能
  • 定义列表：术语和解释的配对列表
  • 脚注：自动编号或手动指定编号
  • 文献引用：与 BibTeX/Zotero 集成，[@smith2020] 格式引用
  • 交叉引用：图表、表格、章节的自动编号和引用
  • LaTeX 数学公式：
    • 行内公式：$...$，如 $\alpha + \beta = \gamma$
    • 块级公式：$$...$$，如 $$\int_{a}^{b} f(x) \, dx = F(b) - F(a)$$
    • 相比 OFM，Pandoc 支持更多 TeX 扩展和公式编号，适合严谨的学术排版
  • 原始 HTML/LaTeX：在 Markdown 中直接插入目标格式的原生代码
  • 多列布局、自定义容器等出版级功能
  • 原生支持导出：一键转换为 PDF、Word、ePub、LaTeX、HTML 等格式

适用场景：

  • 学术论文和学位论文
  • 技术书籍和手册
  • 需要多格式发布的文档（如同时生成网页和 PDF）

代表平台： Pandoc 命令行工具、Zettlr、R Markdown

3.4 方言选择建议

你的需求 推荐方言 理由
刚入门，想快速上手 GFM 兼容最广、文档最多、功能够用
在 GitHub 上写项目文档 GFM 原生支持，功能完全匹配
建立个人知识库 OFM 双向链接和图谱是知识管理的核心
写学术论文 Pandoc 引用、公式、多格式导出一步到位
追求最大兼容性 CommonMark 严格规范，任何解析器都能正确渲染
不知道选哪个 GFM 它是目前的事实标准，几乎所有编辑器都支持

一个重要的提醒： 不同方言之间并不是完全互斥的。你可以先用 GFM 写日常内容，在需要知识管理时切换到 Obsidian，在需要发表论文时导出为 Pandoc 格式。关键是先掌握基础语法（它们在所有方言中都一样）然后学会GFM，再根据场景学习扩展功能。

小结

在上篇中，我们介绍了 Markdown 的核心理念与优势、六大应用场景、上手工具与教程推荐，以及四种主要方言的特性与选择建议。Markdown 不是一款软件，而是一种思维方式。它让你相信：内容永远比排版重要，可迁移性永远比功能丰富重要，纯文本的简洁永远比富文本的复杂重要。
